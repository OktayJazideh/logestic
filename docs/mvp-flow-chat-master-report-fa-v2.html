۱. چشم‌انداز و اصل طلایی (طبق سند معماری کارفرما)

این پروژه فقط اپ حمل بار یا اپ تعاونی نیست. یک زیرساخت دیجیتال برای مدیریت عملیات، تسویه، شفافیت و حکمرانی تعاونی‌های صنعتی است؛ ترکیبی از Logistics Platform + Financial Settlement System + Cooperative OS + Governance Infrastructure.

اصل طلایی پروژه:
«هیچ فرآیند رسمی خارج از سیستم معتبر نیست.»
ماموریت، تأیید، تسویه، عضویت، اعتراض، باسکول — همگی باید داخل سیستم ثبت، تأیید و Audit شوند.

ساختار قدرت (طبق سند)

  • شرکت اپراتور پلتفرم: مالک Core System + Rule Engine + Settlement Engine + Wallet Engine + Audit System. پیمانکار معدن نیست، تعاونی نیست، واسطهٔ پنهان پول نیست — فقط اپراتور شفاف زیرساخت.
  • تعاونی: مجری محلی + معرف خانوار/راننده/ناوگان + بازوی اجتماعی. Settlement را کنترل نمی‌کند، Ruleها را تغییر نمی‌دهد، Audit را حذف نمی‌کند.
  • کارفرما (معدن/کارخانه): ثبت نیاز، ثبت ماموریت، تأیید عملیات، پرداخت هزینه.
  • خانوار: ذی‌نفع اجتماعی، مشاهدهٔ سهم/کیف‌پول/وضعیت تسویه، ثبت اعتراض.
  • مالک ناوگان: درآمد بر اساس عملیات تأییدشده.
  • راننده / اپراتور: مجری عملیات، بدون دسترسی به اطلاعات مالی دیگران.

اصول معماری اضافه‌شده در سند کارفرما

  • Event-Driven Architecture — همهٔ اتفاقات Event هستند.
  • Rule Engine قابل پیکربندی — هیچ Ruleای Hardcode نباشد (درصد سهم، Threshold وزن، دورهٔ Settlement …).
  • Ledger-Based Finance — هیچ Balance مستقیم ویرایش نشود.
  • Idempotency روی عملیات حساس (Settlement، Mission Create، Payment، Weighbridge).
  • Queue + Retry + Reconciliation برای کارهای Async.
  • Soft Delete برای هیچ دادهٔ مهمی delete واقعی نشود.
  • Offline Strategy در اپ راننده برای محیط معدن.
  • Notification در سه کانال: Push / SMS / In-App.
  • KPI & Observability برای راندمان ناوگان، درصد تأخیر، HOLD، failed settlement.
  • Geo-Configurable برای معدن/جاده/روستا/محدوده.

۳. تصمیمات قطعی کارفرما (تبدیل سوال به جواب)

سوال‌های نسخهٔ قبلی این گزارش بسیاری‌اش با سند معماری کارفرما جواب گرفت. این بخش، تصمیم‌های قطعی است و دیگر سوال نیستند:

  • ۱) تعاونی-معدن: فاز ۱ = یک معدن + یک تعاونی. ولی معماری از روز اول باید چندتایی پشتیبانی کند (Geo-Configurable + Multi-Tenant).
  • ۲) فرمول سهم ۱۳٪ خانوار: سهم اجتماعی منطقه است و مستقل از تعداد ماموریت‌های خانوار. فرمول رسمی: community_pool ÷ household_snapshot_count. یعنی پایان ماه، عکس (snapshot) از همهٔ خانوارهای فعال گرفته می‌شود و سهم به‌طور مساوی بین‌شان تقسیم می‌شود.
  • ۳) تعاونی واسطهٔ مالی نیست: پرداخت از پلتفرم مستقیماً به IBAN هر ذی‌نفع (مالک ناوگان / خانوار). تعاونی فقط مجری محلی + معرف عضو است.
  • ۴) Reverse با محدودیت زمان: فقط داخل بازهٔ زمانی محدود + ثبت دلیل + Audit. بعد از آن فقط Adjustment رسمی.
  • ۵) آستانهٔ اختلاف وزن: ۵٪ → HOLD/FLAG اتومات. این Threshold باید configurable باشد (داخل Rule Engine).
  • ۶) حالت پرداخت بانکی: Settlement Lifecycle رسمی: PENDING → CALCULATED → READY_FOR_SETTLEMENT → IN_BANK_QUEUE → SETTLED / FAILED.
  • ۷) عملیات ساعتی: نقش رسمی OPERATOR با START/END + امکان ثبت عکس + توضیح + لوکیشن. CONSULTANT_VERIFICATION گام جدا.
  • ۸) Dispatch: باید کاملاً سیستمی باشد (نه انتخاب دستی). هدف: حذف رانت / حذف رابطه. Marketplace آزاد یا انتخاب ماموریت توسط راننده نداریم.
  • ۹) Mission State Machine: ۹ حالت رسمی: CREATED → ASSIGNED → ACCEPTED → ARRIVED → LOADED → IN_TRANSIT → DELIVERED → VERIFIED → SETTLED.
  • ۱۰) Permission Matrix: ۸ نقش رسمی: ADMIN، OPERATION_ADMIN، COOP_ADMIN، COOP_OPERATOR، CONSULTANT، DRIVER، FLEET_OWNER، HOUSEHOLD.
  • ۱۱) ساختار سه کلاینت: Admin Panel + Driver App + Community App (برای تعاونی و خانوار با Role-Based).
  • ۱۲) Ledger: هیچ Balanceای مستقیم ویرایش نمی‌شود — همه چیز Transaction-Based + Immutable.
  • ۱۳) Idempotency: روی Settlement / Mission Creation / Payment / Weighbridge اجباری.
  • ۱۴) Soft Delete: هیچ دادهٔ مهمی Hard Delete نمی‌شود.
  • ۱۵) Notification: سه کانال Push / SMS / In-App برای ماموریت/Settlement/HOLD/اعتراض/تأیید/پرداخت.
  • ۱۶) Event-Driven: همه‌چیز Event است (mission_created, settlement_generated, payment_failed, vehicle_approved, household_added …).
  • ۱۷) خارج از Scope MVP: AI سنگین، سیستم ملی، Marketplace آزاد، انتخابات، شبکهٔ اجتماعی، چند معدن همزمان، پلاک‌خوان پیشرفته.

۱۰. پیشنهادهای ما (محاوره‌ای)

  • اول پایه‌ها، بعد ویترین. Vision کارفرما خیلی بزرگ است — Event-Driven، Rule Engine، Queue، Reconciliation. اما اگر بخواهیم همه را یک‌دفعه شروع کنیم، هیچ‌کدام تمام نمی‌شود. توصیه: اول DB + Permission Matrix + Mission FSM ۹‌حالته (E0+E1+E2) — این سه باعث می‌شود به یک «نسخهٔ قابل دموی واقعی به یک معدن» برسیم. بعد E4 (مالی) و سپس E5 (زیرساخت).
  • Event Bus در ابتدا in-process باشد. برای MVP لازم نیست Redis/Kafka. یک EventEmitter ساده + جدول events کافی است. وقتی به ۵ معدن رسیدیم، مهاجرت می‌کنیم.
  • Idempotency را زود وارد کنید. هزینهٔ افزودنش الان کم است. اگر تا production نیاید، آن‌وقت کابوس می‌شود.
  • Rule Engine را در همان روز اول راه بیندازیم. چون اگر فردا کارفرما گفت «۸۰/۱۵/۵»، نمی‌خواهیم به ۲۰ جای کد دست بزنیم. حتی اگر فقط یک key (split.owner) در آن باشد، باز ارزش دارد.
  • SMS واقعی را زود وصل کنید. چون کارفرما عملاً با SMS تست می‌کند («راننده هنوز پیام نگرفته!»). اگر قرار است Kavenegar باشد، همان روز اول API Key بگیرید.
  • Community App را پس از تکمیل بک‌اند شروع کنید. اگر همین الان شروع کنیم، هر تغییر روی API هم driver_app و هم community_app را خراب می‌کند. بهتر است یک API Layer پایدار باشد بعد UI.
  • Reconciliation شبانه را با لاگ شفاف بنویسید. این پایهٔ اعتماد کارفرما است: هر شب گزارش «امروز هیچ mismatch نبود / این موارد بود».
  • یک Staging مستقل برای کارفرما بالا بیاید. هر کارت تمام شد، روی Staging مستقر شود، کارفرما خودش با موبایل تست کند.
  • گزارش هفتگی نیم‌صفحه‌ای. «این هفته چی تمام شد، کجا گیر کردیم، هفتهٔ بعد چی کار می‌کنیم». طولانی نباشد، خوانده می‌شود.
  • نقش AUDITOR_READONLY بعداً اضافه شود. برای حسابرس مستقل / نمایندهٔ تعاونی. در سند نبود ولی نیاز خواهد شد.

۱۱. انتقادهای رو راست (محاوره‌ای)

  • سند Vision عالی است ولی Scope بزرگ‌تر از MVP شده. سند می‌گوید «فقط MVP» و در همان زمان «Event-Driven، Rule Engine، Queue، Reconciliation، KPI، Offline، چندکاناله، Community App و …» می‌خواهد. اگر همه را در MVP بریزیم، MVP حداقل ۸ ماه طول می‌کشد. باید با کارفرما توافق کنیم کدام‌ها واقعاً «روز اول لانچ» لازم‌اند (P1) و کدام‌ها «اولین آپدیت بعد از لانچ» (P2). در این گزارش این تفکیک شده ولی کارفرما باید تأیید کند.
  • کد فعلی عمدتاً Postgres است (DB-1 + DB-2). ماموریت، کیف پول، تسویه و audit در DB؛ hourly logs و objections هنوز RAM.
  • چند انحراف از اسپک قبلی هنوز هست. WB-1، HOURLY-1، MISSION-FSM-1، RATE-1، RULE-1، POOL-1، SET-1، REVERSE-1، EVENT-1، QUEUE-1، RECON-1، NOTIF-1 انجام شد؛ باقی موارد P2.
  • «فرمول جدید ۱۳٪» تغییر معماری بزرگی است. از «به نسبت ماموریت» به «pool ÷ snapshot». این کار به migration دیتای قبلی و arrange جدید transaction نیاز دارد. خوبی‌اش: عادلانه‌تر است. سختی‌اش: همهٔ تست‌ها و expectationها باید بازنویسی شوند.
  • عدم وجود «حساب پلتفرم» در مدل فعلی. ۲٪ به wallet با نام platform می‌رود ولی صاحب آن wallet چه کسی است؟ باید یک system_user تعریف شود.
  • تعارض احتمالی بین «Dispatch سیستمی» و واقعیت میدانی. کارفرما می‌خواهد «حذف رانت و انتخاب دستی». اما در محل ممکن است گاهی راننده‌ای مریض باشد و باید جابجا شود. باید endpoint «اضطراری Re-Dispatch» با ثبت دلیل پیش‌بینی کنیم تا قانون شکسته نشود ولی واقعیت هم پاسخ بگیرد.
  • Snapshot ماهانهٔ خانوارها ابهام دارد. اگر در روز ۲۹ ماه یک خانوار APPROVED شد، آیا در snapshot همان ماه می‌آید یا نه؟ پیشنهاد ما: cutoff روز آخر ماه + ساعت ۲۳:۵۹. باید رسماً تأیید شود.
  • یک‌دست نبودن نام‌گذاری در کد و اسپک. کد COOP دارد، اسپک COOP_ADMIN/COOP_OPERATOR. وضعیت ماموریت Flutter اکنون با FSM ۹‌حالته هم‌تراز است؛ سایر mismatchها هنوز بررسی شوند.
  • TEST-1 انجام شد (Vitest + Playwright + CI). اسکریپت‌های tsx test:* قدیمی همچنان مکمل هستند؛ در refactorهای بعدی پوشش Vitest را گسترش دهید.

۱۴. قاعدهٔ «یک چت = یک تسک»

  1. یک چت جدید باز کنید.
  2. فقط پرامپت همان کارت را پیست کنید.
  3. اگر مدل سوال پرسید، جواب بدهید؛ اگر کد تولید کرد، روی branch جدا commit کنید.
  4. پس از تمام شدن، پیامی به ما با عنوان «کارت کد-کارت تمام شد» و لینک PR بفرستید.
  5. ما در گزارش بعدی، آن کارت را حذف می‌کنیم و وضعیت ماتریس را به‌روز می‌کنیم.
هشدار: هرگز دو کارت را در یک چت قاطی نکنید. اگر یک تسک خیلی طولانی شد، آن را به دو تسک بشکنید.
